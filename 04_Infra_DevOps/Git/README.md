# Git 커밋 메세지 규칙

## 1. 기본 구조

커밋 메세지는 변경 내용을 간결하게 요약하기 위해 다음과 같은 형식을 사용한다.

<타입>: <변경 요약> [선택: 상세 내용]


- `타입`: 변경의 성격을 나타내는 키워드 (add, update, fix 등)
- `변경 요약`: 이번 커밋에서 무엇을 했는지 한 줄로 설명
- `상세 내용`: 필요한 경우 구체적인 설명을 추가 (선택 사항)

---

## 2. 자주 사용하는 타입

| 타입 | 의미 | 예시 |
|------|------|------|
| `add` | 새로운 파일이나 기능 추가 | `add: JSP param 정리 문서 추가` |
| `update` | 기존 파일의 내용 수정 | `update: JSP 이벤트 설명 보완` |
| `fix` | 오타나 오류 수정 | `fix: jsp:param 코드 예제 오타 수정` |
| `refactor` | 코드 구조 개선 (동작 변화 없음) | `refactor: 폴더 구조 정리` |
| `remove` | 파일이나 불필요한 코드 삭제 | `remove: 중복된 예제 파일 삭제` |
| `docs` | 문서 관련 수정 | `docs: README.md에 JSP/JSTL 관계 설명 추가` |

---

## 3. 작성 규칙

- 메시지는 **간결하고 명령형 어조**로 작성한다.  
  예: “추가함” 대신 “추가”, “수정함” 대신 “수정”
- 커밋 하나에는 **하나의 목적**만 담는다.  
  (예: 문서 수정과 코드 수정은 별도의 커밋으로 분리)
- 한글로 작성해도 괜찮지만, **일관된 표현**을 유지한다.
- 여러 파일을 수정한 경우, **핵심 변경 사항**만 메시지에 포함한다.

---

## 4. 예시

```bash
add: JSP param과 이벤트 정리 파일 추가
update: JSP param section 내용 보완
docs: README.md에 JSP/JSTL 관계 설명 추가
fix: 이벤트 테이블 정렬 오류 수정
